# Testcase Namer Skill

根据测试用例内容，生成符合规范的**用例名称**和**用例编号**。

---

## 核心规范

### 用例名称（Name）规范

| 规则 | 说明 |
|------|------|
| 长度 | **不超过 40 个字符** |
| 语言 | 推荐英文，中文场景可用拼音缩写或中英混合 |
| 分隔符 | 使用 `_` 分割单词，禁止空格 |
| 特殊字符 | **禁止**使用除 `_` 之外的特殊字符（无 `-`、`/`、`@`、`#` 等） |
| 表达 | 简洁、明确，**禁止模糊语句**（如 test_something、check_it、verify_case） |
| 结构建议 | `[模块]_[操作/场景]_[预期结果]`，例如：`login_valid_cred_success` |

**禁止示例：**
- ❌ `test_login` — 过于模糊
- ❌ `verify_user_can_successfully_login_with_correct_username_and_password` — 超长
- ❌ `login-success` — 含非法字符 `-`
- ❌ `登录成功` — 含特殊字符（中文空格/标点风险）

**合格示例：**
- ✅ `login_valid_user_success`
- ✅ `cart_add_item_exceed_stock_fail`
- ✅ `payment_wechat_timeout_retry`

---

### 用例编号（ID）规范

| 规则 | 说明 |
|------|------|
| 合法字符 | 仅限 `[a-zA-Z0-9_.&-]`，正则：`^[a-zA-Z0-9_.&-]{1,60}$` |
| 长度 | 总长度 **< 45 字符**（在正则允许的 60 内，实际要求更严格） |
| 结构建议 | `[ProjectCode].[Module].[SeqNo]` 或 `[Module]-[Feature]-[SeqNo]` |
| 序号 | 推荐 3 位数字补零，如 `001`、`042` |

**禁止示例：**
- ❌ `TC#001` — 含 `#`
- ❌ `用例_001` — 含中文
- ❌ `TC/LOGIN/001` — 含 `/`

**合格示例：**
- ✅ `PROJ.LOGIN.001`
- ✅ `AUTH-login_valid-001`
- ✅ `TC_CART_ADD_001`

## 工作流程

当用户提供测试用例内容时，按以下步骤执行：

### Step 1 — 解析用例意图

从用例描述中提取：
1. **功能模块**（如：登录、购物车、支付、用户管理）
2. **操作/场景**（如：正常登录、超出库存、超时重试）
3. **预期结果**（如：成功、失败、跳转、报错）
4. **测试类型**（如：正向、负向、边界、性能）

### Step 2 — 生成候选名称

- 将提取要素拼接为 `模块_场景_结果` 格式
- 转为英文（或拼音缩写），全部小写
- 仅保留字母、数字、`_`
- 截断到 40 字符以内

### Step 3 — 生成用例编号

- 若用户提供了项目代码/模块前缀，沿用之
- 否则从模块名推断 2~4 字母缩写作为前缀
- 拼接序号（若用户未指定，从 001 开始）
- 验证：`^[a-zA-Z0-9_.&-]{1,60}$` 且长度 < 45

### Step 4 — 输出标准格式

输出时使用下方「标准输出格式」，并附上简要说明，解释名称/编号各段含义。

---

## 标准输出格式

```
用例名称：<name>
用例编号：<id>

说明：
- 名称解读：[模块] + [操作] + [结果] 的组合
- 编号解读：[前缀含义] + [序号]
- 字符数校验：名称 X 字符（≤40 ✅）；编号 Y 字符（<45 ✅）
```

若用户一次提供多个用例，输出表格：

```
| 序号 | 用例名称 | 用例编号 | 名称长度 | 编号长度 |
|------|----------|----------|----------|----------|
| 1    | xxx      | xxx      | XX ✅    | XX ✅    |
| 2    | xxx      | xxx      | XX ✅    | XX ✅    |
```

---

## 边界处理规则

| 情况 | 处理方式 |
|------|----------|
| 名称 > 40 字符 | 缩写中间词，优先保留模块和结果 |
| 编号 ≥ 45 字符 | 缩短模块缩写，减少序号位数 |
| 含中文 | 转拼音首字母缩写（如"登录"→`login`或`dl`） |
| 含特殊字符 | 替换为 `_` 或直接删除 |
| 用户提供前缀 | 必须沿用用户指定的前缀，不得修改 |
| 描述模糊 | 向用户询问：模块名、操作场景、预期结果三要素 |

## 示例

### 输入示例 1（单用例）

> 测试用户使用正确的用户名和密码登录系统，系统应跳转到首页

**输出：**
```
用例名称：login_valid_credential_redirect
用例编号：AUTH.LOGIN.001

说明：
- 名称解读：login（登录模块）+ valid_credential（正确凭据）+ redirect（跳转结果）
- 编号解读：AUTH（认证模块）+ LOGIN（登录功能）+ 001（序号）
- 字符数校验：名称 33 字符（≤40 ✅）；编号 13 字符（<45 ✅）
```

---
