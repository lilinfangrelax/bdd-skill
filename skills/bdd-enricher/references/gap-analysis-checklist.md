# Gap Analysis Checklist

对已有 `.feature` 文件做缺口分析时，逐类过一遍以下清单。
每一项标注：✅ 已覆盖 / ❌ 未覆盖 / N/A 不适用。

---

## 一、Sad Path — 输入校验类

> 用户提供了无效、格式错误或空白的输入

| 编号 | 检查项 | 示例场景 |
|------|--------|---------|
| SAD-I-1 | 必填字段为空 | 密码为空时提交 |
| SAD-I-2 | 格式非法 | 手机号不是11位数字 |
| SAD-I-3 | 超出长度限制 | 用户名超过50字符 |
| SAD-I-4 | 包含非法字符 | 用户名含特殊符号 |
| SAD-I-5 | 值超出范围 | 年龄填写负数或超过150 |
| SAD-I-6 | 组合字段不一致 | 两次输入的密码不匹配 |

---

## 二、Sad Path — 业务规则类

> 输入格式合法，但违反了业务约束

| 编号 | 检查项 | 示例场景 |
|------|--------|---------|
| SAD-B-1 | 资源已存在（重复创建） | 注册已有手机号 |
| SAD-B-2 | 资源不存在（操作对象缺失） | 查询不存在的订单 |
| SAD-B-3 | 权限不足 | 普通用户访问管理员功能 |
| SAD-B-4 | 状态不允许当前操作 | 已取消的订单不能再支付 |
| SAD-B-5 | 超出数量/金额限制 | 购物车超过99件 |
| SAD-B-6 | 时间窗口过期 | 验证码超时、活动已结束 |
| SAD-B-7 | 依赖项未满足 | 未完成实名认证无法提现 |
| SAD-B-8 | 账号/资源被锁定或封禁 | 被封号的用户尝试登录 |

---

## 三、Edge Case — 边界值类

> 在合法范围的边界处行为是否符合预期

| 编号 | 检查项 | 示例场景 |
|------|--------|---------|
| EDGE-1 | 最小合法值 | 密码恰好6位（最短要求） |
| EDGE-2 | 最大合法值 | 用户名恰好50字符（上限） |
| EDGE-3 | 最小值 - 1（刚好非法） | 密码5位 |
| EDGE-4 | 最大值 + 1（刚好非法） | 用户名51字符 |
| EDGE-5 | 零值 / 空集合 | 购物车为空时结算 |
| EDGE-6 | 单条记录 | 搜索结果只有1条 |
| EDGE-7 | 恰好触发阈值 | 第5次失败触发锁定（不是第4次） |

> 边界值场景适合用 `Scenario Outline + Examples` 合并同类项

---

## 四、业务规则场景（Business Rule）

> 需求中明确说明的计数、时间、状态机转换规则

| 编号 | 检查项 | 示例场景 |
|------|--------|---------|
| BIZ-1 | N 次触发某机制 | 失败5次 → 账号锁定 |
| BIZ-2 | 超时后状态恢复 | 锁定30分钟后自动解锁 |
| BIZ-3 | 状态流转路径 | 订单：待付款 → 已付款 → 发货中 → 已完成 |
| BIZ-4 | 计数重置条件 | 成功登录一次后失败计数清零 |
| BIZ-5 | 角色差异行为 | 管理员和普通用户看到不同的菜单 |
| BIZ-6 | 互斥条件 | 同一优惠券不能叠加使用 |

---

## 五、环境 / 系统异常场景（可选，视团队而定）

> 外部依赖不可用时系统是否优雅降级

| 编号 | 检查项 | 示例场景 |
|------|--------|---------|
| ENV-1 | 网络超时 | 发送验证码请求超时后提示用户重试 |
| ENV-2 | 第三方服务不可用 | 短信服务异常时的降级提示 |
| ENV-3 | 并发写入冲突 | 两个用户同时抢最后一件库存 |

> 这类场景实现成本高，优先级通常低于 Sad Path 和 Business Rule。建议在缺口报告中列为「可选补充」，由用户决策。

---

## 六、如何判断缺口优先级

| 优先级 | 判断标准 | 建议 |
|--------|---------|------|
| 🔴 高 | 需求文档明确提及的失败条件 | 必须补充 |
| 🟠 中 | 业务逻辑推导出的隐含规则 | 建议补充 |
| 🟡 低 | 边界值（+1/-1） | 用 Outline 合并后补充 |
| ⚪ 可选 | 环境/并发/第三方依赖 | 列出供用户决策 |

---

## 七、步骤复用决策树

在生成新 Scenario 的每一个 Given / When 步骤前，先走这棵决策树：

```
该步骤的语义是否在「已有步骤池」中存在？
  ├─ 完全相同 → 直接复用，不改一个字
  ├─ 结构相同、只有参数不同（如手机号换了一个）→ 复用结构，替换参数值
  ├─ 动作相同、条件略有不同 → 考虑将条件参数化，扩展原步骤
  └─ 语义确实不同 → 允许新增步骤
```

**禁止**：为了"看起来更适合这个场景"而重新措辞一个语义已存在的步骤。
