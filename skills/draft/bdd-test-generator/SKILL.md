# BDD 测试用例生成器

将软件需求转化为结构完整、可执行的 Gherkin BDD 测试用例。

---

## 第一步：需求解析

在生成用例前，从用户输入中提取以下信息：

| 维度 | 说明 | 若缺失则 |
|------|------|---------|
| **功能模块** | 这个 Feature 的核心业务目标 | 从需求推断 |
| **主要角色** | 谁在使用这个功能（用户/管理员/系统） | 默认为"用户" |
| **前置条件** | 场景发生前系统的初始状态 | 从业务逻辑推断 |
| **验收标准** | 明确列出的通过/失败条件 | 从需求推断常见断言 |
| **技术约束** | 如字段长度、格式要求、性能指标 | 若有则纳入边界用例 |

如果需求描述过于简短（少于 2 句话），在生成前先向用户确认关键信息。

---

## 第二步：场景覆盖矩阵

对每个功能点，生成以下类型的场景，**按优先级排序**：

### 必选场景（每个功能至少覆盖）
1. **主流程（Happy Path）** — 最典型的成功路径，至少 1 个
2. **主要失败路径** — 最常见的失败原因，至少 1 个
3. **权限/角色边界** — 不同角色的访问差异（若适用）

### 推荐场景（视需求复杂度添加）
4. **边界值** — 最大值、最小值、空值、零值
5. **状态依赖** — 同一操作在不同系统状态下的行为差异
6. **并发/幂等** — 重复操作的结果（若涉及写操作）
7. **数据驱动** — 多组输入用 `Scenario Outline + Examples` 合并

---

## 第三步：Gherkin 写作规范

### 文件结构

```gherkin
# language: zh-CN（中文需求用中文；英文需求用英文）

@标签1 @标签2
Feature: <功能名称，动宾结构>
  作为 <角色>
  我希望 <能力描述>
  以便 <业务价值>

  Background:（可选，若多个场景共享前置条件）
    Given <公共前置步骤>

  @smoke
  Scenario: <场景名，描述完整行为结果>
    Given <初始状态>
    When  <触发动作>
    Then  <可观察结果>
    And   <补充断言>

  @edge-case
  Scenario Outline: <参数化场景名>
    Given <含 <参数> 的前置>
    When  <操作 <参数>>
    Then  <期望结果 <期望>>

    Examples:
      | 参数   | 期望   |
      | 值1   | 结果1 |
      | 值2   | 结果2 |
```

### 步骤编写原则

| 原则 | 正确示例 | 错误示例 |
|------|---------|---------|
| **Given 描述状态，不描述动作** | `Given 用户已登录系统` | `Given 用户点击了登录按钮` |
| **When 只有一个核心动作** | `When 用户提交注册表单` | `When 用户填写表单并点击提交并等待响应` |
| **Then 验证可观察结果** | `Then 系统显示"注册成功"提示` | `Then 数据库中创建了用户记录` |
| **步骤复用，避免重复** | 使用 `Background` 抽取公共步骤 | 每个场景重复写相同 Given |
| **具体而非抽象** | `Then 页面跳转至 "/dashboard"` | `Then 操作成功` |
| **第三人称描述** | `用户点击...` | `我点击...` |

### 标签规范

```
@smoke        — 冒烟测试，核心主流程
@regression   — 回归测试
@edge-case    — 边界条件
@negative     — 异常/失败路径
@api          — 接口层测试
@ui           — 界面交互测试
@wip          — 开发中，暂不执行
@P0 / @P1 / @P2  — 优先级
```

## 第四步：输出格式

### 标准输出（单文件）

直接输出 `.feature` 文件内容，用代码块包裹：

````
```gherkin
# 文件名建议：<模块名>.feature

Feature: ...
  ...
```
````

### 多模块输出

若需求涵盖多个功能模块，分别输出多个代码块，并在每块前注明文件名：

```
📄 login.feature
📄 checkout.feature  
📄 order-management.feature
```

### 附带说明

在代码块之后，简要说明：
- 场景总数及分类统计
- 哪些场景需要测试数据准备
- 哪些场景建议优先自动化

---

## 第五步：质量自检清单

生成完毕后，对照以下清单自检，不合格则修正：

- [ ] 每个 Feature 有且仅有一个业务目标
- [ ] 每个 Scenario 名称能独立说明测试意图
- [ ] Given 步骤无动作动词（点击/输入/提交）
- [ ] When 步骤只有一个触发动作
- [ ] Then 步骤全部可验证、可观察
- [ ] 重复的前置条件已提取到 Background
- [ ] 边界值已用 Scenario Outline 合并
- [ ] 所有场景已打合适标签
- [ ] 无"操作成功"、"系统正常"等模糊断言
